# Siratee Lertsubthaworn — Personal Website

A modern, content-driven personal website built with [Astro](https://astro.build), Tailwind CSS,
and Markdown/JSON content collections. Deploys as a static site to GitHub Pages — no backend
required.

---

## 1. Quick Start (local development)

You need [Node.js](https://nodejs.org) 18.17+ installed.

```bash
npm install
npm run dev
```

Open `http://localhost:4321`. Changes to content files reload automatically.

```bash
npm run build      # builds the static site into ./dist
npm run preview    # serves the production build locally to sanity-check it
```

## 2. Editing content (no code required)

You should almost never need to touch layout or component code to update your information.
Everything editable lives in two places:

### A. `src/data/*.json` — structured facts

| File | Controls |
|---|---|
| `personal.json` | Name, title, elevator pitch, bio, headshot, resume/CV/vCard file paths, logo paths, research interests, languages, hobbies |
| `social.json` | Every social/contact icon shown site-wide. Set `"enabled": false` to hide a platform without deleting it. `"primary": true` shows it in the hero. |
| `skills.json` | Skill categories + levels (used on Home and About) |
| `experience.json` | Work history (About, Resume, CV) |
| `education.json` | Degrees (Home, About, Resume, CV) |
| `certifications.json` | Certifications (About sidebar, CV) |
| `awards.json` | Awards (About sidebar, CV) |
| `services.json` | "How I can help" cards on the Contact page |
| `faq.json` | FAQ accordion items |
| `site.json` | Site-wide settings: pagination size, loading-animation duration/toggle |
| `linkedin.json` | Latest LinkedIn posts shown on the homepage — see note below |

### B. `src/content/*` — long-form content (Markdown)

Each of these is a folder. **Add a new item by adding a new `.md` file** — you do not need to
register it anywhere else.

| Folder | Powers |
|---|---|
| `content/projects/` | Portfolio page + individual project pages |
| `content/blog/` | Blog index + individual posts |
| `content/testimonials/` | Testimonials page + homepage preview |
| `content/publications/` | Publications page + CV |
| `content/resources/` | Resources page |

Every file starts with **frontmatter** (the `---` block at the top) defining structured fields,
followed by free-form Markdown body content. Copy an existing file in the same folder as a
starting template — the required fields are documented in `src/content/config.ts`.

### C. Resume / CV PDFs

Just replace `public/files/resume.pdf` or `public/files/cv.pdf` with your real PDF, **keeping the
same filename**. The Resume and CV pages, and the "Download Resume" buttons site-wide, always
point at these exact paths, so nothing else needs to change.

### D. LinkedIn posts

LinkedIn's API does not allow pulling a personal profile's public post feed into a third-party
site. Two practical options, both configured in `src/data/linkedin.json`:

1. **Manual (default):** add an entry to the `posts` array each time you want to feature a post.
   Takes about a minute.
2. **RSS bridge:** if you'd rather automate it, point `rssUrl` at an RSS feed of your LinkedIn
   activity generated by a bridging service (e.g. RSS.app or similar), set `sourceType` to
   `"rss"`, and wire up a small build-time fetch in `linkedin.json`'s consumer
   (`src/pages/index.astro`) to pull from it. The manual path is wired up out of the box; the RSS
   path needs ~10 lines of fetch code if you want it.

### E. Images

Drop new images anywhere under `public/images/` and reference them by absolute path (e.g.
`/images/projects/my-cover.jpg`) in the relevant frontmatter field. Placeholder SVGs are used
throughout so the site works before you add real photography — swap them out at your own pace.

## 3. Design system

- **Fonts:** Inter (body) + Manrope (display/headings), loaded from Google Fonts in
  `src/styles/global.css`.
- **Color tokens:** defined in `tailwind.config.mjs` — `ink`/`paper` (light mode text/background),
  `midnight`/`cloud` (dark mode), and a `gold` scale used as the single accent color. Change the
  hex values there to re-theme the entire site.
- **Dark mode:** class-based (`dark:` variants), toggled by the button in the nav, defaults to the
  visitor's OS preference, remembers manual overrides in `localStorage`.
- **Loading animation:** `src/components/LoadingScreen.astro`. Controlled by
  `site.json → loadingAnimation` (enable/disable, play-once-per-session, duration).

## 5. Deploying to GitHub Pages

### Option A — GitHub Actions (recommended, already set up)

1. Push this project to a new GitHub repository.
2. In the repo, go to **Settings → Pages → Build and deployment → Source**, and choose
   **GitHub Actions**.
3. Open `astro.config.mjs` and set `site` to your actual URL:
   - User/organization site (`https://<username>.github.io`): leave `base` commented out.
   - Project site (`https://<username>.github.io/<repo-name>`): set
     `site: 'https://<username>.github.io'` and uncomment
     `base: '/<repo-name>'`.
4. Push to `main`. The included workflow (`.github/workflows/deploy.yml`) builds and deploys
   automatically — check the **Actions** tab for progress. Your site will be live at the URL from
   step 3 within a couple of minutes.

### Option B — Manual build + `gh-pages` branch

```bash
npm run build
npx gh-pages -d dist
```

(Requires the `gh-pages` package: `npm install --save-dev gh-pages`.)

### Custom domain

Add a `public/CNAME` file containing your domain (e.g. `siratee.com`), point your DNS at GitHub
Pages per [GitHub's docs](https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site),
and update `site` in `astro.config.mjs` to match.

## 6. SEO checklist (already wired up)

- Per-page meta titles/descriptions via `src/components/SEO.astro`
- Open Graph + Twitter Card tags
- Canonical URLs
- `Schema.org Person` structured data
- `robots.txt` (`public/robots.txt`)
- `sitemap-index.xml` (auto-generated by `@astrojs/sitemap` at build time)
- `rss.xml` blog feed
- Semantic HTML, lazy-loaded images, and `prefers-reduced-motion` support throughout

Run a Lighthouse audit against `npm run preview` before shipping changes — the design system was
built to score well by default (no render-blocking fonts beyond the Google Fonts request, no
layout-shifting animations, keyboard-focus states on every interactive element).

## 7. Extending the site

- **New page:** add a `.astro` file under `src/pages/`. Wrap its content in `<BaseLayout>` and
  pass `title`/`description` props for SEO.
- **New nav link:** edit the `links` array in `src/components/Nav.astro`.
- **New content type:** add a schema to `src/content/config.ts`, create the matching folder under
  `src/content/`, and build a listing/detail page pair (copy the `blog/` or `portfolio/` pair as a
  template).
- **Contact form backend:** the form on `/contact` is a static placeholder (GitHub Pages can't run
  server code). Wire it to a service like Formspree or Getform by pointing the `<form>`'s `action`
  at their endpoint, or swap the JS `submit` handler in `contact.astro` for a `fetch()` call.

## 8. Browser & device support

Built mobile-first with Tailwind's responsive utilities; tested breakpoints cover iPhone/Android
phone widths, tablets (iPad/Android), and desktop. Uses only evergreen CSS/JS — no IE support, no
polyfills needed for current Safari/Chrome/Edge/Firefox.
